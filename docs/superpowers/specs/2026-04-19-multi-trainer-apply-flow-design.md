# Multi-Trainer Apply Flow — Design

**Date:** 2026-04-19
**Phase scope:** Phase 1 of a three-phase multi-trainer rollout. Phases 2 (public trainer directory) and 3 (self-service trainer onboarding + payments) are out of scope for this spec.

## Goal

Transform the single-trainer "Contact" form into a multi-trainer application flow so:

1. Any TRAINER-role user can share a personal referral link (or short code) and receive applications that route directly to them.
2. Clients who already know which trainer they want can apply to that trainer specifically, skipping the platform-wide intake queue.
3. Clients without a preference still apply through a generic path; admin/platform routes them to an available trainer.
4. Once matched, a client sees their trainer's branding on personal surfaces (dashboard, emails, notifications) — not always Brent's.

The invite-only model is preserved: applications produce `ContactSubmission` rows; the chosen trainer (or admin) reviews and issues an invitation. Registration still requires an invitation code.

Explicit non-goals:

- Public trainer directory (Phase 2)
- Self-service trainer onboarding — trainers are still created by admin (Phase 3)
- Per-trainer payment split via Stripe Connect (Phase 3)
- Per-trainer email sender domain verification (Phase 3)

## Section 1 — Data model

Three schema additions, all on models we already have.

### `User` / Trainer additions

Trainer identity lives on the `User` row (role `TRAINER`). New nullable columns:

- `trainerSlug     String?  @unique` — URL slug such as `brent-martinez`. Auto-generated from `name` at first promotion to TRAINER role. Null for non-trainer users.
- `trainerReferralCode  String?  @unique` — 6-char uppercase alphanumeric, `nanoid(6)` with the alphabet `ABCDEFGHJKLMNPQRSTUVWXYZ23456789` (no 0/O/1/I — unambiguous when spoken). Format for display: `MF-4X7K9` (the `MF-` prefix is purely cosmetic; the stored value is `4X7K9` six chars).
- `trainerIsPublic      Boolean  @default(false)` — reserved for Phase 2 directory visibility. Has no effect in Phase 1 except being set to `true` on existing trainers so Phase 2 can flip it later without a backfill.
- `trainerAcceptingClients  Boolean  @default(true)` — drives the "accepting clients" chip and the `/apply/[slug]` behavior (form vs. waitlist).

### `ContactSubmission` additions

- `trainerId   String?` — nullable FK to the `User` the applicant chose.
- `trainer     User?    @relation("TrainerApplications", fields: [trainerId], references: [id], onDelete: SetNull)` — SetNull so deleting a trainer user doesn't cascade-delete historical applications.
- `waitlist    Boolean  @default(false)` — true when the trainer had `trainerAcceptingClients = false` at the time of submission. Admin/trainer UI shows these separately from active applications.
- Index: `@@index([trainerId, status])` to support the "my applications" view on the trainer dashboard without scanning the whole table.

Null `trainerId` means "no preference / platform triage." The admin contacts view shows these in a dedicated `UNASSIGNED` bucket.

### Backfill migration

Because `trainerSlug` and `trainerReferralCode` are `@unique`, the migration that adds them must also populate values for existing TRAINER rows before enforcing the uniqueness.

```sql
-- add nullable columns
ALTER TABLE "users" ADD COLUMN "trainerSlug" TEXT;
ALTER TABLE "users" ADD COLUMN "trainerReferralCode" TEXT;
ALTER TABLE "users" ADD COLUMN "trainerIsPublic" BOOLEAN NOT NULL DEFAULT FALSE;
ALTER TABLE "users" ADD COLUMN "trainerAcceptingClients" BOOLEAN NOT NULL DEFAULT TRUE;

-- populate existing trainers
-- (per-trainer UPDATE statements generated by the migration helper script,
--  or a post-migration seed step that runs once)

-- enforce uniqueness (Postgres unique indexes allow multiple NULLs by default,
-- so no partial-index WHERE clause needed; matches what Prisma emits for @unique
-- on a nullable column)
CREATE UNIQUE INDEX "users_trainerSlug_key" ON "users"("trainerSlug");
CREATE UNIQUE INDEX "users_trainerReferralCode_key" ON "users"("trainerReferralCode");
```

Slug generation rule (applied in code, not SQL):

1. Lowercase the name, replace whitespace with `-`, strip non `[a-z0-9-]` chars, trim leading/trailing dashes.
2. If collision with an existing `trainerSlug`, append `-2`, `-3`, etc. until unique.

Referral code generation rule:

1. `nanoid(6)` with the custom alphabet above.
2. Retry up to 5 times on collision (vanishingly rare given the key space, but cheap).

## Section 2 — Routes and URL structure

| URL | Component | Purpose |
|---|---|---|
| `/apply` | Generic apply page with trainer picker | Top-of-funnel for prospects without a link. Shows search + code-entry + "no preference" options. |
| `/apply/[trainerSlug]` | Direct apply page | Pre-filled with the trainer identified by the slug. Hero is personalized with the trainer's name + photo. |
| `/apply/success` | Confirmation page | Shown after a successful form submit. Confirms which trainer the application routed to, sets expectations on review timing. |
| `/t/[trainerSlug]` | 308 redirect → `/apply/[trainerSlug]` | Reserved for Phase 2 trainer profile pages. Phase 1 just redirects so bio links using the shorter `/t/slug` form still work. |
| `/contact` | 308 redirect → `/apply` | Preserves any existing external links to the old contact page. |
| `/api/trainers/search?q=<query>` | API (GET) | Typeahead search; returns `[{ id, name, slug, photoUrl, initials, acceptingClients }]` filtered by `trainerIsPublic=true`. Limits to 10 results. Rate-limited. |
| `/api/trainers/resolve-code` | API (POST) | Body `{ code: string }`. Returns `{ slug: string }` if found, `{ error: 'not_found' }` otherwise. Rate-limited. |
| `/api/applications` | API (POST) | Existing contact submission endpoint, modified to accept `trainerId` (nullable) and honor the `trainerAcceptingClients` waitlist logic. |
| `/trainer/settings` | Extended | New "Sharing" section renders the referral link, QR code, trainer code, and accepting-clients toggle. |
| `/admin/users/[id]` | Extended | For TRAINER rows, admin can override slug, regenerate referral code, and force pause intake. |

Rate limits on both `/api/trainers/search` and `/api/trainers/resolve-code` are in place to prevent enumeration: 30 req / minute / IP. Results exclude non-public trainers so Phase 2 visibility gating works out of the box.

Pricing CTA change: every `Start 14-day trial` / `See the app` button on marketing pages becomes a single `Apply` button that routes to `/apply` (or `/apply/[slug]` when the user arrived from a trainer profile URL in Phase 2 — Phase 1 always goes to `/apply`).

## Section 3 — Apply-flow UX

### `/apply` (generic)

**Hero:**

```text
APPLY · MARTINEZ/FITNESS
Train with a coach who knows your numbers.
1:1 programming, weekly check-ins, invite-only. 48h review.
```

**Trainer picker card** (above the form):

Three options, vertically stacked with `── or ──` separators:

1. **Search trainers** — debounced typeahead (200ms) against `/api/trainers/search`. Each result is a row: avatar + name + specialty chip + accepting-clients dot. Clicking a row sets the selection state to that trainer and scrolls to the form.
2. **Enter trainer code** — 6-char input with auto-uppercase and spaces stripped. On submit, POST to `/api/trainers/resolve-code`; on success, the page navigates to `/apply/[slug]` (so the locked state takes over). On failure, inline error: "We don't recognize that code. Check with your trainer or search by name above."
3. **No preference — match me** — button that sets the selection state to `{ trainerId: null }` and scrolls to the form.

**Form** appears below the picker with the selection state rendered as a confirmation chip ("Applying as · No preference" or "Applying to · Sarah Jones").

### `/apply/[trainerSlug]` (direct)

- Hero replaces the platform name with the trainer's name and photo. Copy is trainer-driven: `"Train with [Trainer Name]."` plus optional one-liner bio pulled from the Trainer profile (falls back to a default if blank).
- "Applying with" card is a locked confirmation — the trainer's avatar + name, with a small "Not who you meant? [change →]" link that navigates back to `/apply` (clearing the selection).
- If `trainerAcceptingClients = false`, the form is replaced with a waitlist card: same fields, submit button reads `Join waitlist`, submission is flagged `waitlist = true`.
- If the slug doesn't exist: 404 via Next.js `notFound()`.

### Form (shared component)

**Fields** (minimum-viable, aligned to the "maximum first-contact volume" product goal):

- `name` (required, 2-120 chars)
- `email` (required, validated)
- `phone` (optional, E.164-ish, helper text "faster reply if you include it")
- `goal` — chip group, single-select: `get-stronger` | `lose-weight-recomp` | `other`. When `other`, free-text `goalOther` textarea appears.
- `message` (textarea, 500 char limit, optional)
- Hidden: `trainerId` (from selection state)
- Submit button: `Submit application →` (or `Join waitlist` if the trainer paused intake)

**Rate limit:** 5 submissions / hour / IP. Prevents spam, matches existing contact-form limit.

**Above the form — primary direct channel:**

```text
┌─────────────────────────────────────────────┐
│  Fastest reply — text the trainer directly  │
│  [ 📱  Text · (559) 365-2946 ]              │
│  ───────────  or write it out below  ─────── │
└─────────────────────────────────────────────┘
```

When applying to a specific trainer, the phone number becomes that trainer's on-file phone if set (Trainer profile has `phone` — use when present; fall back to the platform number `(559) 365-2946` otherwise).

### `/apply/success`

Renders:

- Confirmation headline: "Application received."
- Body copy explaining the 48h review and invitation-by-email flow.
- The trainer they applied with (avatar + name) — or "Platform triage" if no trainer was selected.
- The email the trainer will reach them at.
- Two secondary CTAs: `Text Brent` and `Instagram` (or the specific trainer's channels when applicable).

The page must be accessible via a tokenless URL (no session needed) because the applicant isn't a registered user yet. To prevent someone from hitting `/apply/success` cold and seeing stale data, the page reads one-time success state from the HTTP response of the POST (encoded in a short-lived signed cookie, not in the URL — no PII in URLs). If the cookie is absent, the page redirects to `/apply`.

## Section 4 — Trainer settings "Sharing" panel

Lives in `/trainer/settings` as a new card beneath the existing account/password sections. A client component because everything here is interactive.

### Displayed controls

- **Personal link**
  - Full URL: `<domain>/apply/<trainerSlug>` rendered as a read-only text field.
  - Copy-to-clipboard button, 2-second "COPIED ✓" confirmation.
  - Small helper text: "Add to Instagram bio, TikTok, business cards."

- **QR code**
  - 120×120 preview generated server-side at request time via the `qrcode` package. Encodes the full apply URL.
  - `Download PNG` and `Download SVG` buttons, each fetches `/api/trainers/qr?format=png` (or `svg`) which regenerates on demand. Accent color inherits `--mf-accent`.

- **Trainer code**
  - 6-char code displayed in a monospace pill with the cosmetic `MF-` prefix (so `4X7K9` displays as `MF-4X7K9`; the actual DB value is the 6 chars).
  - Copy button.

- **Accepting-clients toggle**
  - Two-state: "● Accepting new clients" (green) or "Paused — waitlist only" (amber).
  - Toggle hits `PATCH /api/trainers/me` with the new bool; optimistic UI.
  - Helper text explains the behavior ("Your link still works; prospects see the waitlist card instead of the form").

### Admin-only extensions

Rendered inside `/admin/users/[id]` when the user is a TRAINER. Admin sees:

- Override slug (free-text field; validates uniqueness and re-checks format)
- Regenerate referral code (with "Are you sure?" confirm — invalidates any printed QR codes)
- Force pause intake (overrides trainer's own toggle; shows "Paused by admin" in the trainer UI)

## Section 5 — Per-client trainer branding

Once a client is matched with a trainer (`User.trainerId` set on the client row), the client's own dashboard surfaces display their trainer, not Brent.

### Single source of truth

A client-side hook:

```tsx
useAssignedTrainer() → {
  id: string
  name: string
  initials: string   // "SJ" for "Sarah Jones"
  photoUrl: string | null
} | null
```

Fetches once per mount from `/api/clients/me/trainer` (authenticated, returns the current user's assigned trainer summary). Null if the client has no assigned trainer (possible while in the matching window).

### Surfaces that change

- **Client DesktopShell brand block** — replaces `MARTINEZ/FITNESS · CLIENT · V4` with the trainer's name and "YOUR COACH" label. Brand mark shows trainer photo or initials.
- **Client workout session headers** — "Programmed by [Trainer Name]"
- **Client notifications** — "[Trainer Name] sent you a message" replaces the generic "New message"
- **PR achievement emails** — display name in the email body reads the trainer, not hardcoded "Brent Martinez Fitness." Sender domain and Reply-To stay on the platform in Phase 1 (Phase 3 introduces per-trainer sender verification).
- **Celebration coach-note avatar + name** — initials from `useAssignedTrainer()`; coach label reads "COACH [First Name]" for that trainer's clients.

### Surfaces that stay platform-branded

- All marketing pages (`/`, `/about`, `/pricing`, `/programs`, `/contact`-redirect)
- Auth pages
- Admin console
- Invoices, receipts, legal
- Resend `From:` display name (stays `Brent Martinez Fitness <noreply@...>` in Phase 1)
- Trainer directory `/trainers` (Phase 2)

### Fallback behavior

If `useAssignedTrainer()` returns null (matched but not yet assigned, or data mid-flight), the shell and all surfaces fall back to platform branding — no broken layout, no empty string. Every component reads a branding prop that defaults to the platform and accepts a trainer override.

## Architecture summary

- **One hook** (`useAssignedTrainer`) drives all per-client branding.
- **One shared form component** for `/apply` and `/apply/[slug]` — differs only in initial selection state.
- **Three new API routes** (search, resolve-code, QR), one extended route (`/api/applications`).
- **Schema delta** lives in a single migration that adds four columns to `users`, three columns + one index to `contact_submissions`, and a backfill step for existing trainers.
- **No new dependencies beyond `qrcode`** (MIT, ~150kB, zero known CVEs at the time of writing).

## Testing strategy

No test framework is set up in this repo. Verification for this feature uses:

- `npx tsc --noEmit` — type safety must be clean
- `npm run build` — production build must succeed
- Manual smoke flows (documented in the implementation plan):
  1. Existing Brent trainer slug + code populated correctly after migration.
  2. `/apply/brent-martinez` renders trainer hero, locked selection, working form submit.
  3. `/apply` search returns Brent, code-entry with Brent's code redirects to `/apply/brent-martinez`.
  4. Submit with `trainerId=null` creates an unassigned submission visible in admin triage.
  5. Pausing intake on Brent's settings flips the apply page to waitlist mode.
  6. A client assigned to a (hypothetical second) trainer sees that trainer's branding in their dashboard.
  7. `/contact` redirects to `/apply`; `/t/brent-martinez` redirects to `/apply/brent-martinez`.
